# A股量化交易系统 — 审计报告

> 审计日期：2026-06-13 | 审计范围：全系统（数据层、策略引擎、回测引擎、风控、实盘、Web API）

---

## 一、致命缺陷（必须立即修复）

### 1.1 基本面数据完全缺失

**现状：** `fundamentals` 表为空（0行），`stock_info` 表中 PE/PB/市值 全部是占位符（PE=20、PB=3、市值=100亿，81只股票一模一样）。

**影响：**
- 8因子策略中 ROE、profit_growth、revenue_growth、accrual_ratio 四个因子全部失效
- 价值策略、质量策略完全没有意义
- PE分位判断（位置策略）使用的PE是假数据

**修复：** 从 AKShare 拉取基本面数据填充 `fundamentals` 表，并实现按报告期的点-in-time查询（数据库 `get_financial_for_date()` 已经写好了，只缺数据）。

### 1.2 前视偏差（Look-Ahead Bias）

**现状：** `data/fetcher.py` 的 `build_market_data_by_date()` 对每只股票只拉取一次财务数据，然后应用到回测窗口内所有日期。

**举例：** 回测2023年1月时，使用的PE/ROE是2025年Q4的数据——你在2023年不可能知道这个数据。

**修复：** 使用 `fundamentals` 表存储按报告期的时间序列，回测时通过 `get_financial_for_date(ts_code, trade_date)` 获取该日期实际可用的最新报告。

### 1.3 Web API 无任何鉴权

**现状：** `server.py` CORS 全开（`allow_origins=["*"]`），实盘下单、启动/停止服务等接口无任何密码或token保护。

**风险：** 同一局域网内任何人都可以访问 `/api/live/submit_order` 下单。

**修复：** 至少加一个 API Key 或简单的密码验证中间件。

## 二、高风险问题（建议近期修复）

### 2.1 交易日历缺少中国节假日

**现状：** `trade_calendar` 表有8797条记录，但 `is_open=0` 的为0条。周末通过"不在表中"来排除，但春节、国庆、五一等中国假期全部被当成交易日。

**影响：** 回测会在春节7天休市期间模拟交易；实盘会在假期尝试扫描。

**修复：** 调用 AKShare 的交易日历接口或使用 `pandas_market_calendars` 获取准确的A股交易日历。

### 2.2 分钟数据完全为空

**现状：** `minute_bars` 表0行数据。

**影响：**
- 日内反转策略完全无法使用
- 无法做日内做T分析
- 实盘无法做分钟级监控

**修复：** 写定时任务在交易时段采集5分钟K线数据。AKShare的 `stock_zh_a_hist_min_em` 接口支持。

### 2.3 实盘数据链路不完整

**现状：** 此前的修复（`_build_market_data_from_db`）解决了从数据库加载历史衍生指标的问题，但：
- 数据库中的PE/PB是假数据（见1.1），策略中的价值因子不可用
- 实时行情 API 只给 OHLCV，不包含任何基本面信息
- `DataFetcher.get_financial_data()` 使用硬编码行号解析AKShare返回（`data/fetcher.py:347`），一旦AKShare改格式就会静默返回错误数据

### 2.4 风控缺少连续亏损熔断

**现状：** `broker/risk_manager.py` 有单日亏损限制（-2%），但没有：
- 连续亏损天数熔断（连亏5天应立即暂停）
- 最大回撤熔断（从权益峰值回撤超过一定幅度）
- 单月最大亏损限制
- 交易频率限制（防止算法失控频繁交易）

### 2.5 配置文件缺乏校验

**现状：** `BacktestConfig.from_dict()` 直接构造数据类，没有验证：
- 止损率可能被设为500%
- 最大持仓数可能为0或负数
- `stop_loss_rate > move_stop_rate`（止损应比移动止盈更紧）

## 三、中等问题（影响可控但应修复）

### 3.1 幸存者偏差

**现状：** 股票池在获取时过滤了ST、退市、科创板、北交所。这意味着：
- 回测中不会出现"买了一只后来ST退市的股票"这种情况
- 实际交易的亏损风险被低估

**修复方案：** 不需要保留所有退市股，但至少应在回测报告中注明"本回测已排除ST/退市股票，真实风险可能更高"。

### 3.2 因子引擎降级路径不稳定

**现状：** `strategy/ai_strategy.py` 在没有模型文件时回退到 `FactorEngine`，但 `FactorEngine` 依赖 `statsmodels` 库，而这个库不在 `requirements.txt` 中。如果用户只装了 `requirements.txt`，AI策略会静默崩溃。

### 3.3 相对路径风险

**现状：** 多处使用 `./data_cache` 相对路径。如果从不同目录启动服务（如通过 systemd、计划任务），所有缓存文件会写到错误位置。

**修复：** 使用相对于 `settings.py` 所在目录的绝对路径。

### 3.4 数据库无审计日志

**现状：** `data_log` 表完全为空。无法追溯数据何时更新、更新了多少条、是否有异常。

### 3.5 券商连接失败静默降级

**现状：** `live_server.py:58-65` — QMT连接失败时静默切换到模拟盘，用户可能以为在实盘交易。

**修复：** 降级时发送明确通知（控制台+web），并在API状态中反映真实连接状态。

## 四、低优先级（架构优化建议）

### 4.1 策略注册表缺乏版本管理

当前11个策略全部硬编码在 `strategy/__init__.py` 中。建议改为配置文件驱动，方便新增/禁用策略，同时记录策略版本号以追踪回测结果对应的策略版本。

### 4.2 回测结果缺乏持久化存储

回测结果存在内存字典 `results` 中，服务重启即丢失。建议存入SQLite或文件，支持历史回测结果的查询和对比。

### 4.3 缺少策略基准对比

回测只输出绝对收益，没有与基准（沪深300、中证500）的对比。无法判断策略的超额收益是否来自市场beta。

### 4.4 缺少参数敏感性分析

参数优化器存在但功能有限。建议增加单参数敏感性分析——固定其他参数，遍历单个参数看收益变化曲线。

### 4.5 前端缺少移动端适配

`live.html` 有 `mobile.html` 版本，但 `kline_vis.html` 和 `app.html` 在手机上基本不可用。
